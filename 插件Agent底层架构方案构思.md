# 思源笔记「试验插件」Agent 底层架构方案构思

> 文档版本：v0.1  
> 日期：2026-05-15  
> 依据：《思源笔记 AI Agent 集成设计稿》v0.4、《调研 Notion Custom Agents》、《调研思源 AI 插件》；本文仅讨论 **官方试验插件** 在工程上的 **Agent 运行时与编排层** 选型，不替代后续内核 API 定稿。

---

## 一、问题边界与共同前提

### 1.1 试验插件要回答什么

试验阶段建议聚焦设计稿中的 **A 轨「笔记助手」**：用户主动发起的多轮对话、工具调用、工作集与写确认、基础运行日志。B 轨（定时、事件触发、后台队列）复杂度高，**不宜作为底层架构试验的第一约束**，但工具协议与授权模型应 **预留** 与 B 轨共用，避免日后推倒重来。

### 1.2 所有方案共享的产品约束（来自设计稿与调研）

| 约束 | 说明 |
|------|------|
| 授权优先 | 显式工作集；Instructions / 对话里的链接 **不** 自动等于授权（对齐 Notion） |
| 可观测 | 工具序列、错误、粗 token 日志；为后续 Activity 面板留结构化事件 |
| 写路径 | 大段改写倾向 **Diff / propose_edit**；高危工具（如 SQL）默认关或二次确认 |
| 交付形态 | 插件 **仅能** 使用文档化插件 API；与社区 Copilot / Sisyphus 等可长期并存 |

### 1.3 评价维度（用于对比各方案）

- **耦合度**：与思源内核 HTTP / 插件 API 的绑定方式是否集中、可测试。  
- **安全面**：API Token、模型 Key 所在进程；是否易做「工具前闸门」与审计。  
- **演进成本**：未来迁到「原生实现」或「官方 MCP 网关」时，Tool 协议能否平移。  
- **开发效率**：首版能跑通 A 轨闭环的工程量与依赖栈风险。  
- **生态位**：与现有社区形态（进程内工具、MCP 网关、iframe 外壳）是 **重复** 还是 **互补**。

## 二、方案总览

| 方案代号 | 一句话 | LLM 与编排所在位置 | 典型参考 |
|----------|--------|-------------------|----------|
| **A** | 进程内单体 | 插件前端（渲染进程）内完成循环 + 调内核 API | Copilot、sy-f-misc GPT 子模块 |
| **B** | 插件内双端（Renderer + Server） | 编排可在前端或 Node 侧；内核访问统一经本机回环 | Sisyphus 的 Vite 双构建思路（未必暴露 MCP） |
| **C** | 外置 Host + MCP / HTTP | 模型与 Agent 循环在 Cursor / CLI / 自研服务；插件 = 网关 + 权限 + 轻 UI | Sisyphus、设计稿「官方 MCP 网关」远期 |
| **D** | 外壳 + 块上下文协议 | 插件几乎不调模型；Dock iframe + 拖拽 MIME 传递块上下文 | AI CLI Bridge |
| **E** | 混合：进程内编排 + 可选 MCP 工具合并 | 主循环在插件内；工具列表 = 内置 Tool + 动态 MCP Client | siyuan-agent |

以下各节展开 **运行时拓扑、数据流、适用与不适用**，并给出试验阶段的落地建议。

---

## 三、方案 A：进程内单体（前端直连模型 + 直连思源 API）

### 3.1 拓扑

```text
用户 → 插件 UI（同渲染进程）
        → OpenAI 兼容 HTTP（模型）
        → 思源内核 HTTP / 插件封装 API（工具）
```

Agent 循环（解析 tool_calls → 执行工具 → 拼消息历史）全部在 **插件前端** 实现；工具为 TS 函数 + JSON Schema，内部调用现有 `fetch` / 封装 `request`。

### 3.2 优点

- **路径最短**：与社区 Copilot 能力对齐速度快，适合验证「Tool 清单、工作集、写确认」产品逻辑。  
- **无额外进程**：不依赖用户安装 Node CLI 或单独起服务。  
- **调试简单**：浏览器 DevTools 即可覆盖 UI + 网络。

### 3.3 缺点与风险

- **密钥与配额**：模型 Key 通常落在插件存储；需明确加密存储与「不向笔记内容泄露 Key」的边界。  
- **长任务阻塞 UI**：若不做 Worker 或分片 yield，重检索 + 多轮工具可能卡顿（可用 `requestIdleCallback`、工具并发策略等缓解，但根因仍在主线程）。  
- **与「官方 MCP 网关」远期形态**：若未来要把工具执行迁到受控子进程，需抽象一层 **ToolExecutor 接口**，避免业务逻辑与 `fetch` 细节缠死。

### 3.4 试验建议

作为 **P0 功能验证的默认基线**：先跑通 A 轨闭环，再把「Tool 定义 + 授权检查 + 审计日志」沉淀为 **与 UI 解耦的模块**，为方案 B / E 复用。

## 四、方案 B：插件内双端（Renderer + 本地 Server Bundle）

### 4.1 拓扑

```text
插件 UI（渲染进程） ←→ localhost:插件端口（Node / Bun 等 Server 入口）
        Server → 模型 API（可选：统一代理、流式转发）
        Server 或 Renderer → 思源内核 API
```

构建形态可参考 Sisyphus：**双 target**（`renderer` + `server`），`onload` 拉起本机 HTTP 服务。与方案 C 的差别在于：**编排与模型调用仍由官方插件掌控**，Server 侧重 **网关、流式聚合、密钥隔离、限流**，而非把大脑交给任意第三方 Host。

### 4.2 优点

- **凭据与策略上移**：模型 Key、请求日志、工具白名单可在 Server 层统一实施，渲染进程只持会话令牌或短期 JWT（若自建 mini 协议）。  
- **便于接流式与重试**：SSE / chunked 在 Node 侧处理比在纯前端少一层 CORS 与浏览器差异。  
- **对齐「官方标准化 MCP 或等价 RPC」**：Server 可演进为 **受控 MCP Host** 或内部 gRPC，而不必一开始就全量 MCP。

### 4.3 缺点与风险

- **安装与端口**：防火墙、端口占用、多工作区多实例冲突；需清晰错误提示与自动端口探测。  
- **打包与签名**：桌面端对本地监听的安全感知；文档需说明 **仅本机回环、默认关闭外网**。  
- **工程量**：高于方案 A，适合 A 轨验证通过后 **第二迭代**。

### 4.4 试验建议

当产品明确要写 **「运行日志 + 配额 + 密钥不落渲染进程」** 中的任意两条时，优先考虑从 A **迁移到 B**，而不是直接跳到全外置 C。

## 五、方案 C：外置 Host + MCP / HTTP（薄插件网关）

### 5.1 拓扑

```text
外部 Agent Host（Cursor / CLI / 自建）
        → MCP 或 HTTP（插件或内核暴露的受控端点）
        → 思源数据面（块 / 文档 / 检索）
插件 UI：配置连接、笔记本级 Token、工具开关、运行审计展示
```

插件核心能力是 **「思源侧 API 网关 + 权限 + 审计」**；模型与多步推理在外部。对齐 Notion 叙事：**MCP 不具备触发能力**，触发仍来自用户或 A 轨会话（设计稿已强调）。

### 5.2 优点

- **与「用户自选模型与编排栈」一致**：不重造 Cursor / Codex Web 等体验。  
- **进程隔离最强**：恶意提示词导致的「乱调工具」仍须在网关层治理，但 LLM 实现细节不在思源进程内。  
- **与社区 Sisyphus 关系清晰**：官方试验可走 **「规范 + 参考实现 + 更强默认安全策略」**，减少碎片化网关。

### 5.3 缺点与风险

- **依赖外部服务就绪**：用户环境变量多，排障链路长（Host、端口、Token、TLS）。  
- **体验碎片化**：Chat 主体会落在外部 UI，思源侧 Activity 与「工作集」需额外协议同步。  
- **试验 A 轨闭环成本高**：若目标是验证「思源内一体化助手」，C 常作为 **并行轴** 而非唯一轴。

### 5.4 试验建议

适合 **「官方 MCP / HTTP 契约」** 与 **企业治理** 的预研；可与方案 D 组合（外部 Web UI + 官方网关）。与方案 A 共享 **同一套 Tool Schema 与授权模型**，仅在 Executor 实现上区分「进程内 fetch」vs「转发给 MCP」。

## 六、方案 D：iframe 外壳 + 块上下文协议（无内置 Agent 循环）

### 6.1 拓扑

```text
Dock iframe（用户配置的 URL）
        ← 拖拽 MIME（text/siyuan-block-id 等）← 思源编辑器
插件：几乎不实现 LLM；可选配 Sisyphus 供 iframe 页内 MCP 使用
```

### 6.2 优点

- **集成成本最低**：最快验证「块上下文进入外部 Agent」的产品手感。  
- **能力上限交给外部**：适合跟设计稿中「AI CLI Bridge 升格为受信任面板 + MIME 规范」路线对齐。

### 6.3 缺点

- **授权、工作集、写确认、审计** 难以在思源内形成闭环，除非外部 Host 主动对接官方网关 API。  
- **iframe 能力边界**：与编辑器深度协同（定位块、Diff）受限。

### 6.4 试验建议

作为 **并行试验渠道**（尤其是多模型 UI），而非「官方 Agent 唯一底层」。若做，建议把 **MIME 与「显式工作集」** 写进同一套产品文档，避免用户误以为「拖入即授权全库」。

## 七、方案 E：混合编排（内置 Tool + 可选 MCP 客户端合并）

### 7.1 拓扑

```text
插件内 Agent 循环（同 A 或 B）
        → Tool 注册表：内置思源工具（必选路径）
        → 可选：MCP Client 拉取远端工具 → 合并进同一 tool 列表
        → 子 Agent / 降权工具集（可选）：控制 token 与误写面
```

对应 siyuan-agent 已验证的路线：**LangChain 非必须**，核心是 **统一 Tool 接口 + 动态工具列表 + 子 Agent 限权**。

### 7.2 优点

- **一条产品线两种增量**：默认开箱即用（内置工具）；进阶用户接 MCP 扩展。  
- **对齐设计稿 P3**：「插件贡献工具」与「官方 MCP 网关」可共用 Tool 元数据（风险等级、写确认策略）。

### 7.3 缺点与风险

- **依赖与复杂度**：MCP Client 版本、传输兼容、工具名冲突需 registry 规范。  
- **测试矩阵膨胀**：内置工具 × MCP 工具 × 多模型。

### 7.4 试验建议

在方案 A 的 **Tool 注册表** 设计时预留 `source: 'builtin' | 'mcp' | 'plugin-extension'` 与 **工具级策略**，避免后期合并时全局重构。

## 八、横向对比与组合策略

### 8.1 矩阵（简）

| 维度 | A 进程内单体 | B 双端 Server | C 外置 Host | D iframe 外壳 | E 混合 |
|------|-------------|---------------|------------|---------------|--------|
| 首版工程量 | 低 | 中 | 中（网关硬） | 极低 | 高 |
| 密钥隔离 | 弱 | 强 | 最强 | 不适用 | 中～强 |
| A 轨一体化体验 | 好 | 好 | 分散 | 弱 | 好 |
| 与官方远期 MCP 网关 | 需抽象 Executor | 易演进 | 即目标形态之一 | 间接 | 高度一致 |

### 8.2 推荐组合（试验期）

1. **主路径**：**方案 A** 跑通 A 轨（工作集、写确认、日志、Tool 协议）。  
2. **安全与工程化升级**：沉淀 **ToolExecutor + 审计总线** 后，评估 **方案 B**（密钥与流式代理）。  
3. **生态与标准**：并行文档化 **方案 C** 的 HTTP / MCP 契约（可与社区网关互补）。  
4. **UI 试验**：**方案 D** 低成本验证外部 Agent；与设计稿「MIME 规范」统一。  
5. **能力扩展**：在 A 或 B 上叠加 **方案 E** 的 MCP 合并与子 Agent，对照 siyuan-agent 与 Notion「工具级开关 + 写默认确认」。

## 九、对内核 / 插件 API 的共性诉求（跨方案）

无论采用 A～E 哪一种 **运行时** 拓扑，下列 **接口与数据契约** 越早统一，试验越能向「原生或官方插件」平滑迁移：

1. **稳定 Tool 层描述**：名称、JSON Schema、风险标签（读 / 写 / SQL）、默认确认策略。  
2. **工作集序列化**：与设置 API、`conf.AI` 演进兼容的存储形状（设计稿第五节 ABI 考量）。  
3. **Diff / 写回**：`propose_edit` vs `apply_small_patch` 等与编辑器协同的语义（对齐 AI Assistant 调研结论）。  
4. **运行事件结构**：触发源（用户 / 会话 id）、工具调用 id、耗时、错误码，供未来 Activity 与团队审计。

---

## 十、结论

试验插件的 **Agent 底层** 不必一开始就选定唯一拓扑；更稳妥的路线是：**以方案 A 验证产品与 Tool 协议** → **用同一套抽象接方案 B 或 C 满足安全与生态** → **用方案 E 扩展 MCP 与子 Agent** → **方案 D 作为低门槛 UI / 外部 Host 试验入口**。  

该路线与设计稿中「A 轨优先、B 轨预留、原生与插件双交付、MCP 与 Diff 协同」一致，也与社区已有工程样本（Copilot、Sisyphus、siyuan-agent、AI CLI Bridge）形成可对照的 **演进梯度**，便于在内核排期中分阶段 API 化。
