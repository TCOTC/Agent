AI 与 LLM 基础

这一节建立共同语言：当你在后端或插件里「调模型」时，实际在交换什么数据结构，以及「Agent」与「普通聊天」差在哪里。

大语言模型（LLM）是什么

大语言模型是一种根据「前文」预测「下一个词元（token）」的概率模型，通过海量文本训练后，能在对话、摘要、代码生成等任务上表现良好。 工程上你很少直接运行模型权重，而是调用推理服务（本地或云端），例如 OpenAI、Azure、DeepSeek、Ollama 等提供的 HTTP API。

Token 与上下文窗口

Token 是模型处理文本的最小单位，可能是半个汉字、一个英文词根或标点。计费和长度限制通常按 token 计算。 上下文窗口 是单次请求里模型能「看到」的最大 token 数，包括系统提示、历史消息、工具返回等。超出窗口需要截断、摘要或换用更长窗口的模型。

补全（completion）与聊天（chat）

早期接口是「给一段前缀，模型续写」；现代产品几乎统一用 Chat Completions 形态：消息列表里每条有 role（如 system、user、assistant、tool），模型学习在对话结构里生成下一条 assistant 消息。

系统提示（system prompt）

system 消息用于描述助手身份、安全规则、输出格式、工具使用约束等。本插件在 agentLoop.ts 里用 buildSystemPrompt 注入「仅能访问工作集」等规则，属于典型的系统提示工程。

工具调用（function / tool calling）

除了返回自然语言，模型还可以返回结构化的 tool_calls：每个调用包含工具名与 JSON 参数字符串。 客户端（你的插件）负责真正执行工具，把结果以 role: "tool" 的消息写回对话，再请求模型继续推理。 这一「模型提议 → 本地执行 → 结果写回」的闭环，就是本插件 runAgentLoop 的核心。

温度（temperature）与确定性

temperature 越高，采样越随机；越低则越接近「每次都选概率最高的续写」。本插件在 openaiClient.ts 里使用较低温度（0.2），偏向稳定、可重复的助手行为。

OpenAI 兼容 API

许多供应商实现与 OpenAI 相同 URL 与 JSON 字段的接口（例如 /v1/chat/completions），便于客户端共用一套请求代码。 本插件只依赖这一兼容层，因此可替换为任意兼容端点，只要支持 tools 与 tool_calls 语义即可。

流式输出（streaming）

SSE 或 chunked 响应可以边生成边显示，改善首字延迟。本 demo 为简化实现使用非流式 JSON 一次返回；产品化时可改为 stream: true 并处理增量事件。

从「聊天」到「Agent」

若模型只输出文本，用户得到的是「建议」；若模型能循环调用工具并根据结果再决策，就具备「在环境里采取行动」的特征，工程上常称为 Agent 回路。下一章《Agent 主流范式》会展开不同回路形态；本插件实现的是其中最常见的一种：带工具的多轮聊天循环。
