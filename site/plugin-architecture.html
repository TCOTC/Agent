本插件架构

从思源插件入口到模型与内核：分层理解「谁负责 UI」「谁负责循环」「谁负责调 HTTP」「谁负责权限」。

总览：四层结构

  1. 插件壳（src/index.ts）：实现思源 Plugin 生命周期，注册 Dock、设置项，加载 / 保存持久化数据。
  2. Dock UI（src/dock.ts）：拼接 DOM、处理按钮与输入、维护内存中的对话列表与工作集展示，触发 Agent 循环。
  3. Agent 循环（src/agent/agentLoop.ts）：组织消息数组、调用 LLM、消费 tool_calls、写审计事件；是「编排层」。
  4. 工具与内核（builtinTools.ts + kernelExecutor.ts）：把模型意图翻译成思源内核 REST 调用，并做工作集校验与写确认。

数据流（一次用户发送）

用户点击「发送」
  → dock 内 runSend 读取设置（API Key、baseUrl、model）
  → runAgentLoop({ llm, worksetRootIds, messages, userText, signal, onAudit })
       1. 追加 user 消息；构造 system + 历史 convo
       2. 循环（无固定轮数上限；用户可通过 AbortSignal 中止）：
            openAIChatCompletion → POST chat/completions（带 tools）
            若无 tool_calls → 追加 assistant 文本 → return
            若有 tool_calls → 追加 assistant（含 tool_calls）到 convo 与 messages
            对每个 call：runBuiltinTool → kernel.post("/api/...")
            每个结果以 role=tool 追加
  → dock 刷新消息区与审计日志

工作集（workset）与安全边界

工作集是用户显式选中的文档根块 ID（rootID）列表，存于 STORAGE_KEY_WORKSET（磁盘键名 agent-workset）。 只读工具在访问块前会通过 ensureRootInWorkset 校验该块所属文档根是否在工作集中；读文档则要求传入的 id 本身必须是工作集内的根 ID。这样即使模型「想」读别的文档，也会被工具层拒绝。

KernelExecutor 抽象

KernelExecutor 接口只暴露 post(url, body)，默认实现 createFetchSyncKernelExecutor 使用插件内 postKernelJson（标准 fetch POST JSON，不经思源 fetchSyncPost）。未来若改为独立 Node 服务、MCP 或 WebSocket，只需替换实现并注入 runAgentLoop({ kernel: ... })，编排层可保持不变。

审计（Audit）与可观测性

AuditEvent 描述 LLM 请求 / 响应、工具调用与结果、用户消息预览等。dock.ts 里 pushAudit 将其格式化为运行日志，便于排查「模型调了哪个工具、耗时多少、是否失败」。

与思源核心的关系

插件运行在思源前端宿主环境，通过内核 HTTP API 访问笔记本体数据；LLM 调用则发往用户配置的公网或自建兼容端点。 因此安全模型是双端点：一端是「笔记数据」（受工作集 + 确认框约束），另一端是「模型供应商」（受 API Key 与用户协议约束）。
