Agent 主流范式

「Agent」不是单一标准，而是一族让模型与环境反复交互的工程模式。下面列出常见范式，并说明本插件落在哪一类。

1. 工具增强聊天（Tool-augmented chat loop）

模型在每一轮可以返回普通文本，或返回一组 tool_calls；运行时执行工具，把 tool 消息追加进上下文，再调模型。 循环直到没有工具调用或达到轮次上限。优点是实现直接、易与现有 Chat API 对齐。本插件即采用此范式（见 runAgentLoop）。

2. ReAct（推理 + 行动）

论文形态的 ReAct 常在输出中交替「思考句」与「行动句」（如调用搜索）。现代 API 则往往把「思考」藏在链式消息或专用字段（例如某些模型的 reasoning_content）里。概念上仍属于：观察环境 → 推理下一步工具或回答。

3. 规划—执行（Planner / Executor）

先由模型或规则层生成步骤列表，再由执行器逐步完成。适合长任务、需要人类审批的流程。本 demo 未单独拆分规划器，但你可以在上层加「先输出计划再执行」的系统提示或第二模型。

4. 多 Agent（协作或分工）

多个不同系统提示或不同工具的「子助手」通过消息总线、主管模型或固定 handoff 规则协作。扩展思源集成时，可按功能域（检索、写作、排版）拆 Agent，但要注意延迟与成本。

5. RAG（检索增强生成）

在回答问题前，从向量库或搜索引擎取回相关片段放入上下文。本插件用「工作集 + 全文搜索工具」实现显式范围内的检索，而不是隐式向量库；两者可并存：工具负责取数，RAG 负责embedding 召回。

6. MCP（Model Context Protocol）

MCP 是 Anthropic 推动的一种工具与资源暴露协议，把「有哪些工具、如何调用」从应用里抽离到标准服务。 本仓库 types.ts 里 ToolSource 已预留 "mcp"，表示未来可把部分工具从内置改为 MCP 服务而不改 Agent 循环形状。

7. 人机协同（Human-in-the-loop）

在写库、删数据、执行 SQL 等高风险操作前弹窗确认，是企业场景常见要求。本插件对写工具与 SQL 使用 confirmPromise， 用户拒绝后工具返回 user_cancelled，模型在后续轮次应友好说明，体现典型的 HITL 设计。

安全提示：工具能力 = 模型能间接触达的权限。务必结合最小权限（本插件用工作集限定文档根 ID）、 确认框与审计日志，避免「模型被诱导越权读写」。

与本插件的对应关系小结

范式 本插件中的体现
工具增强聊天 runAgentLoop + openAIChatCompletion + runBuiltinTool
RAG / 检索 siyuan_search_blocks、siyuan_read_doc（范围受工作集路径限制）
HITL confirmPromise 写入确认、SQL 每次确认
MCP 类型预留，当前未接 MCP 服务
