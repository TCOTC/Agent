# 思源笔记 AI Agent 集成设计稿

> 文档版本：v0.4  
> 日期：2026-05-14  
> 依据：根目录《调研 Notion Custom Agents》《调研思源 AI 插件》及思源内核现有 AI 配置能力（`conf.AI`、设置 API 等）的交叉归纳；**非**对 Notion 或社区插件的逐条对标承诺。

---

## 一、背景与目标

思源笔记已具备面向「单次补全 / 自定义操作」的 AI 能力（内核侧 OpenAI 兼容配置、编辑器内 AI 菜单等），社区插件已形成 **进程内多工具 Agent**（Copilot、siyuan-agent）、**外置 MCP 网关**（西西弗斯）、**iframe 外壳**（AI CLI Bridge）、**块级 Diff 编辑**（AI Assistant）等多元形态。

本设计稿回答：**若由官方在思源中系统化集成「与笔记交互的 AI Agent」**，可落地哪些能力、如何分层、与现有内核 / 插件生态如何衔接。

**产品目标（建议表述）**

- 把「问一问、改一改」升级为「在明确授权与可观测前提下，对笔记本 / 文档 / 块执行多步推理与工具调用」。
- 与本地优先、数据自托管的思源价值观一致：**默认最小权限、写操作可确认、运行可审计**。

**当前实施范围（重要）**

- **自动化 Agent**（下文的 **B 轨**：定时、文档 / 块事件触发、后台常驻类能力）涉及触发器、任务队列、事件钩子、配额与异常恢复等，**整体复杂度高**。
- **现阶段不纳入实现计划**：下文仍保留 B 轨作为产品与架构的**远期参照**，便于工具协议、授权模型一次性设计得可扩展；**近期落地以 A 轨「笔记助手」为主**（用户主动发起的多轮对话与工具调用）。

---

## 二、产品心智：双轨 Agent

借鉴 Notion「通用助手 vs Custom Agents」的分工，建议在思源采用 **双轨** 表述，降低用户把自动化误解为「全库上帝模式」的预期风险。

| 轨道 | 名称建议 | 启动方式 | 典型场景 |
|------|----------|----------|----------|
| **A** | **笔记助手**（随写随用） | 用户主动打开侧栏 / Tab、或从选区 / 块发起 | 续写、摘要、单次问答、基于当前上下文的轻量多步操作 |
| **B** | **自动化 Agent**（可订阅） | 定时、文档 / 块事件、（远期）外部 Webhook / 同步事件 | 周期性日报、标签 / 属性整理、新文档模板化、跨笔记本汇总 |

两轨可 **共享同一套工具协议与授权模型**，仅在 **触发器、默认权限上限、配额策略** 上区分：B 轨默认更保守（例如写操作一律确认或需单独勾选）。**当前版本仅推进 A 轨交付；B 轨不实现，以下涉及触发器与后台调度的描述均为预留设计，不作为近期排期承诺。**

## 三、思源场景下的功能构思

以下按「用户价值 → 与思源数据模型的契合点」组织，便于排期拆分。

### 3.1 上下文与「工作集」

- **当前工作集**：编辑器焦点文档、选中块、最近打开的 Tab、用户拖入侧栏的块 / 文档引用（与 Copilot 交互习惯对齐）。
- **显式工作集**：用户在 Agent 设置里勾选笔记本 / 文档 / 查询保存的 SQL 视图范围；**自然语言或对话里出现的链接 / ID 不自动视为授权**（对齐 Notion「Instructions 里粘贴链接不等于授权」）。
- **子 Agent / 降权探索**：对大笔记本先只读检索、再缩小到少数文档深入写（参考 siyuan-agent 子 Agent 思路），控制 token 与误写面。

### 3.2 工具能力（对内 API 的一等封装）

在社区插件已验证的需求基础上，建议官方抽象 **稳定 Tool 层**（名称与粒度可对齐 Copilot 分类，但由内核或官方模块统一审计）：

| 域 | 能力示例 | 备注 |
|----|----------|------|
| 块 / 文档 | 读内容、插入、更新、移动、属性（含数据库块） | 数据库块需与现有 AI 行为约束一致（参见 changelog 中「AI 继续写入数据库块」等演进） |
| 检索 | 全文 / 块类型 / 路径 / SQL（若开放） | SQL 类工具需 **高风险标记 + 默认关闭或二次确认** |
| 结构 | 笔记本树、文档大纲、反向链接 | 利于「先理解再写」 |
| 编辑器 UI | 打开文档、定位块、高亮（权限需低于静默改库） | 与 `openTab` 等能力协调，避免多插件重复污染全局 |
| 导出 / 附件 | 只读优先；写文件系统单独授权 | 与本地工作区安全模型一致 |

**原则**：读与写分离配置；写操作支持 **按工具 / 按笔记本** 的「自动 / 每次确认 / 禁止」三档（对齐 Notion MCP 侧「写类默认确认」的思路）。

### 3.3 触发器（B 轨核心，**暂不实施**）

> **说明**：本节描述的是 **B 轨自动化 Agent** 所需能力。**当前阶段不开发**；保留供日后若重启 B 轨时对照需求与内核钩子设计。

思源无 Slack 一类内置 IM，触发器宜 **从本地与笔记生命周期** 出发设计：

1. **定时**：每日 / 每周在指定时刻对「授权工作集」执行指令（如生成日记索引、周回顾）。
2. **笔记事件**（需内核或同步层暴露钩子）：新建文档、块更新、特定属性变化、日记创建等；支持简单过滤（路径前缀、标签、文档类型）。
3. **手动仅运行一次**：与 A 轨共用，便于调试指令（**此项随 A 轨即可支持**，不等同于完整 B 轨）。

**说明**：事件触发比纯聊天对 **配额与后台任务队列** 压力大，需与「治理、配额与商业化留白」一节所述策略一起设计；这也是 B 轨暂缓的重要原因之一。

### 3.4 与「块级编辑 + Diff」的协同

调研指出 **AI Assistant** 的「选区 → 流式 → Diff → 接受 / 拒绝」闭环风险面更清晰。建议官方 Agent：

- 对 **正文大段改写** 优先走 **Diff 预览写回** 路径，而非静默整篇替换。
- Agent 工具层可区分：`propose_edit`（生成补丁 + UI 确认）与 `apply_small_patch`（小范围、用户可配置是否免确认）。

### 3.5 MCP 与「外置大脑」

- **官方标准化 MCP 或等价 RPC**（受控端口、笔记本级 token、与西西弗斯思路对齐）：把「模型与编排」放在 Cursor / CLI / 自研宿主，思源侧做 **网关 + 权限 + 审计**，减轻每个插件重复封装内核 HTTP。
- **边界**：MCP 在 Notion 叙事中 **不具备触发能力**；若未来提供 MCP 与 **B 轨**（届时再评估），应在文档中写清 **触发始终由内核算子或官方调度器发起**，避免用户以为外置服务能「推送」执行任意写操作。**在 B 轨未实现的阶段**，以用户或 A 轨会话发起的工具调用为主，边界相对简单。

### 3.6 模型与配置

- 延续内核 **OpenAI 兼容** 配置，扩展为 **多配置档**（不同 Agent / 不同用途使用不同 base URL / 模型 / Key），与现有 `conf.AI` 演进兼容。
- 可选 **Auto 路由**（按任务类型或启发式选模型）作为进阶能力，非第一版必需。

### 3.7 协作与同步场景（中长期）

若未来强化「工作空间成员」或「多端同步冲突」：

- Agent 配置 **版本历史与回滚**（对齐 Notion Version history 的运营需求）。
- 运行日志中记录 **设备 / 用户 / 触发源**，便于团队排查「谁让 Agent 改了这篇笔记」。

## 四、界面与信息架构（建议）

| 区域 | 作用 |
|------|------|
| **Chat** | 对某一 Agent 或「默认助手」对话；支持附加工作集、单轮调试指令。 |
| **Activity / 运行日志** | 触发来源、工具调用序列、错误、token 粗估；便于 MCP 与失败排查。 |
| **Settings** | 指令（Instructions）、工具与访问范围、模型、写确认策略；**触发器、版本历史入口** 属 B 轨或中长期能力，当前可不提供或仅占位。 |

与现有编辑器 AI 入口的关系：**A 轨可复用或嵌入侧栏**。**B 轨** 若以设置 + 日志为主，可避免与普通写作菜单挤在同一层级（**B 轨未实施前以 A 轨界面为主**）。

### 4.1 界面风格与 DOM 约定

- **视觉与交互**：Chat、运行日志、设置等与 Agent 相关的界面，应与当前思源主界面 **一致的体验语言**（间距、字号层级、图标与按钮语义、侧栏与 Tab 行为等），避免「外挂感」过强的独立皮肤。
- **类名与 DOM 结构**：优先 **沿用思源既有惯例**——复用主应用已采用的 **CSS 类名**、**布局结构** 与 **DOM 层级习惯**（例如与现有设置面板、Dock、对话框、表单控件等实现对齐），新组件也应按同一套结构组织节点，便于主题联动与无障碍策略统一。
- **主题**：**浅色 / 深色** 及 CSS 变量体系须随思源主题切换正确表现；主应用样式迭代时，Agent 界面应同步兼容，不依赖硬编码色值破坏一致性。
- **原生与插件**：无论第五节选型为原生还是官方插件，**上述约定均适用**；插件实现不得在视觉上自成体系，而应在插件 API 允许范围内尽量贴近主界面 DOM / 类名实践。

## 五、官方原生实现 vs 官方插件实现

同一套产品能力（A 轨助手、Tool 协议、写确认等）在工程上可走两条交付路径：**写进主应用 / 内核的原生实现**，与 **以官方维护的插件包形态发布**。二者不是互斥关系（例如原生承载编排与审计，插件只做实验 UI），但边界与成本不同。

| 维度 | 官方原生实现 | 官方提供插件实现 |
|------|--------------|------------------|
| **运行与信任边界** | 与桌面端 / 内核、编辑器同一进程或受控子系统，属应用内置能力 | 与社区插件相同的插件运行时；官方仅表示「发行主体与质保」不同 |
| **前端能调用的能力** | 可接入**未对第三方公开的**内部模块（编辑器状态、Protyle 私有协议、焦点同步、未文档化的 UI 组件等），按内核发版节奏演进 | 插件前端 **只能使用思源对插件专门暴露的插件 API**（文档化入口，如请求内核的封装、Dock / 命令 / 设置面板注册等）；**不能**依赖未暴露的编辑器内部实现，否则视为越权或脆弱耦合 |
| **数据与笔记访问** | 可直接走内部调用链，也可统一走 HTTP / LSAPI；易与全局策略（配额、审计、Diff）对齐 | 通常经 **已暴露的 HTTP / 插件 API** 访问块与文档；能力上限受 API 覆盖面与版本化约束 |
| **安全与 ABI** | 安全模型随内核整体设计；破坏性变更在主干内消化 | 暴露面小、**ABI 稳定性要求高**；新能力往往要先「API 化」再下放给插件 |
| **发版与裁剪** | 随思源主版本；渠道（如应用商店）规则一并适用 | 可独立版本号、可选安装或按渠道下架；适合试验功能、区域策略差异 |
| **界面与 DOM** | 与主应用同源样式与组件，易保持完全一致 | 须在插件沙箱内实现，仍应 **对齐思源类名、DOM 结构与主题变量**（参见 **4.1**），避免独立设计体系 |
| **典型用途** | 默认开启的笔记助手、与编辑器强绑定的 Diff / 选区上下文 | 可选组件、A / B 试验、需弱耦合或快速迭代的侧栏形态 |

**核心差异（摘要）**：插件形态下，**前端仅能使用官方为插件预留并文档化的 API**；原生实现可落在「插件 API 之外」的内部接口上。产品立项时需先决定能力落在哪一侧：若强依赖编辑器内部状态且短期无意 API 化，倾向 **原生**；若强调隔离、可选装与契约稳定，倾向 **插件**，并把缺口反馈为 **新的正式插件 API** 需求。

## 六、与现有内核 / 插件生态的衔接

官方 Agent 由 **原生** 还是 **官方插件** 交付，见 **第五节**；本节从内核配置、社区共存与安全三方面说明与生态的衔接。

1. **内核**  
   - 在现有 `conf.AI` 与设置 API 上扩展：**Agent 列表、授权范围序列化** 等；**触发器元数据** 随 B 轨再定。  
   - **统一任务队列**（定时 + 事件）为 B 轨配套，**当前不实现**；近期可由 A 轨会话内调度即可。

2. **插件**  
   - 官方 Agent 与社区 Copilot / siyuan-agent **可长期并存**；通过 **Tool 注册扩展点** 让插件贡献只读工具或带徽章的高风险工具（默认关闭）。  
   - **AI CLI Bridge** 模式可升格为「受信任面板 + 块拖拽 MIME 规范」，降低试验第三方 UI 的成本。

3. **安全**  
   - 提示词注入、恶意笔记诱导 Agent 滥权：文档层告知风险；产品层 **缩小默认授权 + 写确认 + 日志**；技术层对「从笔记中解析出的 URL / 指令」与「用户配置的 Instructions」分通道权重（细则可单独出安全设计）。

## 七、治理、配额与商业化留白

- **配额维度**：近期以 **A 轨交互**（按用户 / 会话 / 可选按工作空间）为主即可。**B 轨后台** 任务更易放大 token，待 B 轨立项后再区分计量策略。  
- **耗尽行为**：A 轨可提示增配或限流；**「暂停 B 轨」** 等策略在 B 轨存在后再设计（对标 credits 用尽暂停的思路，具体商业模式由产品决定）。  
- **企业 / 团队**：**「仅管理员可创建 B 轨 Agent」** 等与 B 轨绑定，当前从略。

## 八、分阶段落地建议（路线图草案）

| 阶段 | 范围 | 价值 |
|------|------|------|
| **P0** | A 轨：统一 Tool 协议 + 侧栏多轮对话 + 工作集 + 写确认 + 基础日志 | 与社区插件能力对齐但体验与审计统一 |
| **P1** | 块级 Diff 与 Agent 写回打通；子 Agent / 只读预检索 | 控制质量与费用 |
| **P2** | ~~B 轨：定时 + 简单文档事件；配置版本历史~~ **（暂缓，不纳入当前路线图）** | 工作流自动化；复杂度高，日后单独立项评估 |
| **P3** | 官方 MCP 网关形态定型；插件贡献工具；团队权限 | 生态与协作 |

## 九、结论

在思源中集成 AI Agent 的合理路径是：**以授权与可观测性为先**，用 **双轨产品** 区分即时助手与自动化 Agent（**后者暂不实现，仅作架构预留**），用 **稳定 Tool 层 + 可选 MCP** 衔接内核与外部宿主，用 **Diff 与写确认** 守住编辑体验与数据安全；**原生与官方插件两条交付路径**（第五节）决定前端能否触及内部实现或仅能依赖文档化插件 API。**近期执行焦点是 A 轨「笔记助手」**，避免在触发器与后台调度上过早分散工程投入。社区已有实践（Copilot 工具拆分、西西弗斯 MCP、siyuan-agent 子 Agent 与定时任务等）为官方方案提供了可直接对照的工程样本；本设计稿可作为需求评审与架构拆分的起点，具体 API 与数据表设计需与内核版本节奏对齐后再定稿。
