> 文档版本：v1.0  
> 日期：2026-05-29  
> 目的：记录思源 Agent 插件在多会话并行、输入框/停止按钮与会话状态绑定时，可参考的外部设计与推荐演进方向；供后续重构与 Code Review 对照。

## 一、背景与问题

插件侧栏支持多条 `ChatSession`（对话历史、切换、未发送草稿 `composerDraft`）。用户期望：

- 在会话 A 流式输出时切换到会话 B，**A 可在后台继续**；
- 当前可见会话的 **发送 / 停止按钮、输入框草稿** 与 **该会话** 一致；
- 工具确认、流式渲染不要串到错误会话；
- **模型选择** 不宜全局共用，以免破坏 Provider 侧 prefix / KV cache，导致 Token 成本上升。

当前实现（`AppShell.ts`）在闭包内用 `sessionRuns`、`runningSessionId`、`getActiveAgentSession()` 等修补，与 **单套 Composer DOM + 单 agent 全局指针** 存在结构性张力。本文档归纳 pi、Zed 的做法及对本插件的可借鉴结构。

## 二、本插件现状：状态维度

至少四类状态叠在同一 `AppShell` 闭包中：

| 维度 | 存储 | 含义 |
|------|------|------|
| **视图会话** | `sessions.activeId` | 侧栏选中、消息列表与 Composer 当前展示哪条对话 |
| **数据会话** | `ChatSession`（`messages`、`composerDraft`、`mode`…） | 每条对话的持久化数据 |
| **运行会话** | `sessionRuns: Map<sessionId, { ctl, abortAgent }>` | 哪些会话里 Agent 正在跑（可多个） |
| **全局 UI** | 单个 `composerEditor`、`btnSubmit`、`elMessages` | 全 Shell 仅一套输入区与聊天 DOM |

辅助指针（易与上表混淆）：

- `runningSessionId`：最近一次 `runSend` 的会话 id，**不是**「所有在跑会话」的集合；
- `agentRunMessages`：最近一次 run 绑定的 `messages` 引用；
- `getActiveAgentSession()`（`agentLoop.ts`）：全局唯一 agent 实例 + `activeAgentSessionDepth`。

### 2.1 已对齐的行为（近期修补）

| 行为 | 机制 |
|------|------|
| 切换会话时草稿不串 | `switchToSession` → `flushComposerDraft` → 改 `activeId` → `restoreComposerDraft` |
| 停止按钮跟「当前可见会话是否在跑」 | `isActiveSessionStreaming()` = `sessionRuns.has(activeId)` |
| 后台 run 写消息 | `agentLoop` 内 `syncMessages` → 启动时的 `p.messages` |
| 停止打到正确 agent | 每会话 `onRunReady` → `abortAgent` |
| 切回正在跑的会话恢复流式 DOM | `resumeStreamRenderIfNeeded()` + `messageSync` 保留 tool 参数阶段 `_streaming` |

### 2.2 仍不对称或存在风险

| 项目 | 现状 | 风险 |
|------|------|------|
| 模型 / 思考开关 | 全局 `settings` | 换会话共用模型，不利于 KV cache；与 pi / Zed **每会话模型** 不一致 |
| 消息二次同步 | `syncAgentToChatMessages` + `getActiveAgentSession` | 多 run 时可能同步到错误 `messages` |
| 工具确认 | `pendingConfirmResolvers` 全局 | 多 run 时确认 UI / toast 可能串台 |
| 聊天 DOM | 仅渲染 `activeId` | 后台会话无实时 DOM（数据仍更新），切回需 `renderMessages` |
| 删除正在跑的会话 | 侧栏删除 | 未必 `abortSessionRun` |

## 三、pi 的设计（Agent 底座）

详见 [Agent 底座](./Agent底座.md)。与「多会话并行」相关的结论：

### 3.1 `pi-coding-agent` 产品层：**单活跃会话，切换即 teardown**

`AgentSessionRuntime.switchSession` / `newSession` 会 `teardownCurrent()` → `session.dispose()` → 再 `createRuntime()` **新建** `AgentSession` + `Agent`。

**含义**：在 CLI / 单面板产品里，**不支持**「A 在输出时切到 B 且 A 继续在同一个 UI 进程里跑」。切换等于停掉旧 agent 连接。

### 3.2 `pi-agent-core` 库层：**可多实例，偏编排 / 沙箱**

[Discussion #3337](https://github.com/earendil-works/pi/discussions/3337)：企业场景在每个 sandbox 里跑一个 `Agent` 类，上下文序列化到存储，预期每租户 **几十～上百并发 run**。这是 **多进程 / 多实例**，不是单 UI 内多 Tab 并行。

`toolExecution: "parallel"` 指 **同一次 turn 内多个 tool 并行**，不是多个聊天会话并行。

### 3.3 模型与会话 id（KV cache 相关）

| 机制 | 说明 |
|------|------|
| `Agent.sessionId` | 传给 Provider，用于 **prompt / session cache 亲和**（见 pi-agent-core README） |
| 每会话 model | `SessionManager` / JSONL 中 `appendModelChange`，恢复会话时 **从会话文件读回 model** |
| 全局默认 | 仅作 **新建会话** 的默认值 |

**结论**：pi 刻意保持「一条对话 ≈ 固定 model + 稳定 sessionId + 连续 messages」。**换模型会破坏 cache 前缀**；插件若用全局 `settings.model` 发送所有会话，与 pi 相反，有 Token 成本风险。

## 四、Zed 的设计（侧栏多 Thread 并行，**主要借鉴对象**）

参考路径：

- `refer/zed/crates/agent_ui/src/agent_panel.rs`
- `refer/zed/crates/agent_ui/src/conversation_view/`（含 `thread_view.rs`）
- `refer/zed/crates/agent_ui/src/thread_metadata_store.rs`
- `refer/zed/crates/agent/src/thread.rs`

### 4.1 核心数据结构

```text
AgentPanel
├── base_view: BaseView              # 当前可见：某个 ConversationView（或 Terminal / 未初始化）
├── draft_thread: Option<ConversationView>   # 「新建对话」草稿槽
├── retained_threads: HashMap<ThreadId, ConversationView>  # 后台保留的完整视图实体
└── terminals: ...                   # 终端类 Agent 另轨
```

| 概念 | 含义 |
|------|------|
| `ThreadId` | 侧栏列表中的对话 id（本地 UUID） |
| `session_id` | ACP / Provider 会话 id；**首条消息发出后** 才写入 metadata（draft 阶段为 `None`） |
| `ConversationView` | 一条 thread 的 **整块 UI + 状态**（消息区、输入框、模型选择器等） |
| `AcpThread` / `agent::Thread` | 该 thread 背后的连接与生成任务 |

**关键**：后台在跑的 thread 保留 **整棵 `ConversationView` 实体**（含 `cancel` 句柄），而不是只保留消息数组。

### 4.2 切换 thread：保留运行，不 dispose

`set_base_view` 换视图前调用 `retain_running_thread`：将离开的 `ConversationView` 放入 `retained_threads`。

注释（`agent_panel.rs`）：

> Clear the active view, **retaining any running thread in the background**.

切回 thread：`load_agent_thread` 从 `retained_threads.remove(thread_id)` **提升**为 `base_view`，**不重建**连接。测试 `test_background_thread_promoted_via_load` 覆盖：A 生成中 → 打开 B → A 后台 → 再 load A → 仍为同一 `session_id`。

### 4.3 停止按钮：绑定 ThreadView，非全局

`ThreadView::cancel_generation` → `self.thread.update(|thread, cx| thread.cancel(cx))`。每个视图持有自己的 `Entity<AcpThread>`，避免「停 B 却 abort 到 A」。

### 4.4 模型与 cache

- Native thread 从 `DbThread.model` 恢复 `SelectedModel`（`agent/src/thread.rs`）。
- `ThreadView` 上有 `model_selector`，`current_model_id()` 读 **当前 ConversationView** 的模型。
- `ThreadMetadata.session_id` 与 Provider 会话稳定绑定；草稿 promotion 后才分配 `session_id`。

### 4.5 运行状态与内存

- `ThreadStatus::Idle | Generating`：按 **该 view 自己的 thread** 显示。
- `cleanup_retained_threads`：对 **Idle 且可 reload** 的 retained view 做 LRU 淘汰；**Generating 的保留**。

## 五、三方对照

| 维度 | pi coding-agent | Zed Agent Panel | 思源 Agent 插件（现状） |
|------|-----------------|-----------------|------------------------|
| 多会话并行（单 UI） | 否（switch = dispose） | **是**（`retained_threads`） | 半支持（`sessionRuns` + 单 DOM） |
| 每会话 UI 实体 | 单 Session | **每 Thread 一个 ConversationView** | 单 Composer + 单 `elMessages` |
| 停止 | `session.abort()` | **`thread.cancel()` per view** | `sessionRuns` + `abortAgent` |
| 模型绑定 | 每 session JSONL | **每 thread DB + selector** | 全局 `settings.model` |
| Provider session | `agent.sessionId` | **`ThreadMetadata.session_id`** | 无稳定 per-session id |
| 切换会话时 agent | 重建 | **保留实体** | 后台继续写 `p.messages` |
| 流式 DOM（非当前会话） | N/A | 保留在 retained view 内 | 仅持久化数据，切回再渲染 |

## 六、推荐的可借鉴结构（思源插件）

不必照搬 Rust / GPUI，建议 **状态机同构**：

```text
SessionRegistry（概念层，可落在 AppShell 或独立模块）
├── activeSessionId: string
├── draft: { composerDraft, ... }              # 对应 Zed draft_thread
├── retained: Map<sessionId, SessionRuntime>
│     └── SessionRuntime
│           ├── messages: ChatMessage[]         # 已有 ChatSession.messages
│           ├── modelId: string                 # 新增，持久化到 ChatSession
│           ├── providerSessionId?: string      # 新增，利于 cache / 恢复
│           ├── run: { abortCtl, abortAgent }   # 对应 sessionRuns 条目
│           └── uiCache?: { streaming flags }   # 可选，切回时恢复流式路径
└── visible: 仅 activeSessionId 绑定 composerEditor + elMessages
```

### 6.1 切换会话流程（目标顺序）

1. `flushComposerDraft()` → 写入 **旧** `activeId` 的 `composerDraft`
2. 若旧会话正在跑 → `retained.set(oldId, runtime)`（或仅保留 run + messages，见下）
3. `activeId = newId`
4. `restoreComposerDraft()` → 恢复 **新** 会话输入框
5. `syncSubmitRunning()` → 停止按钮 = `retained.has(activeId)` 或当前 run
6. 若新会话在 `retained` → `promote` 并 `renderMessages` / `resumeStreamRender`
7. 侧栏可对 `sessionRuns.has(id)` 显示「运行中」角标（Zed 侧栏 metadata 有 `updated_at` / 状态）

### 6.2 停止 / 发送

- **发送**：`runSend` 只 `abort` **同 sessionId** 的旧 run（已实现），不要误杀其他会话。
- **停止**：`abortSessionRun(activeId)`，调用该会话 `run.abortCtl` + `run.abortAgent`；**禁止**仅依赖 `getActiveAgentSession()`。
- 工具确认回调写消息时必须带 `sessionId`，或从 run 句柄反查 `messages` 引用。

### 6.3 模型与 KV cache（建议 P0）

| 项 | 建议 |
|----|------|
| 数据模型 | `ChatSession.model: string`（及可选 `thinkingEnabled`） |
| 全局设置 | 仅作 **新建会话默认值** |
| 发送 | `runAgentLoop` 使用 **当前会话** 的 model，禁止用全局 settings 覆盖已在跑的会话 |
| Provider session | 每条会话生成稳定 `providerSessionId`（UUID），在 DeepSeek 请求中携带（若 API 支持） |
| Composer 模型下拉 | 改 `getActive().model` 并 `persistSessions`，而非只写 `settings` |

### 6.4 产品策略二选一（避免混用）

| 策略 | 行为 | 对齐 |
|------|------|------|
| **A. 互斥 run（偏 pi）** | 切换会话时 abort 旧会话；实现简单，无后台并行 | pi coding-agent |
| **B. 保留 run（偏 Zed）** | `retained` Map + 每会话 `abortAgent`；需 RunRegistry、确认带 sessionId | Zed `retained_threads` |

**不要**混成「后台多 run + 全局 `getActiveAgentSession` + 单 DOM」，这是当前 bug 类问题的根源。

若选 **B**，DOM 仍可为单套（思源限制），但逻辑上须 **per-session run 句柄** 与 **按 sessionId 的流式调度**（`scheduleStreamRender(originSessionId)`，仅 `originSessionId === activeId` 时改 DOM）。

## 七、演进优先级（建议）

### P0（稳态，改动面小）

1. `ChatSession.model`（+ 发送时用会话 model；Composer 下拉写会话）
2. `sessionRuns` 合并为 `SessionRunHandle { ctl, abortAgent, messages, sessionId }`，删除冗余的 `agentRunMessages` / 弱化 `runningSessionId`
3. `syncAgentToChatMessages` 仅通过 `messages` 引用或 `sessionId` 定位目标，禁止多 run 时依赖 `getActiveAgentSession`
4. 删除正在跑的会话时调用 `abortSessionRun(id)`
5. 侧栏「运行中」指示

### P1（多 run 可靠）

6. `agentLoop`：`Map<sessionId, AgentSession>` 或 run 句柄传入所有 UI 回调
7. `pendingConfirm` / inline action 带 `sessionId`
8. `scheduleStreamRender` 按会话队列或独立 `streamRaf` 登记

### P2（体验）

9. 切到后台 run 的会话时 toast：「另一对话仍在运行」
10. 可选：确认浮层不依赖消息 DOM（全局 modal + sessionId）
